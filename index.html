Audit log (debug + admin)

    Administrator gate

    Before running intent shaping, you must unlock the operator session with your private key and passphrase. This keeps execution mode and history behavior under a local admin session instead of a fixed account value in the UI.

    The service does not ship with any fixed operator email identity; only operator-boundaries created from your current session are used.

    To operate the service, the key is required on demand: private key format must be JWK (RSA / RSA-OAEP-256) or @github variant with decrypt capability.

    Quick operator tutorial

    Run these steps in order from the main page once per tab:

    1. Set your operator identity and unlock admin session with a private key + passphrase.
    2. Keep the same identity in the same tab while shaping to keep the session owner valid.
    3. Select a surface and input intent, then run shaping.
    4. Copy or export payload logs for audit and evidence.
    5. Operator test run:
      1. Try run without unlock: should stay blocked.
      2. Unlock with a short passphrase: should fail.
      3. Unlock with private key + 8+ chars: should succeed and enable run.
    6. Use route aliases:
      • /mcp/client
      • /mcp/server
      • /api/openai/v1/chat/completions
      • /openaiapi, /opeaiapi, /oprenaiapi
    7. For a quick operation check:
      1. Reload page → press Run intent shaping without unlock (must be blocked).
      2. Unlock with passphrase under 8 chars (must fail).
      3. Paste an admin private key in RSA-OAEP-256 or @github form + 8+ chars passphrase (must succeed).
      4. Run shaping and confirm all three boxes (policy / envelope / response) update.

    ユーザ向け軽量セルフチェック

    1. ページを開いた直後、実行前に Run intent shaping を押して、ブロックされることを確認。
    2. 同じ operator でパスフレーズを 7 文字など短く入れて Unlock すると失敗することを確認。
    3. RSA-OAEP-256/@github 形式の private key(JSON) と 8 文字以上の passphrase で unlock できることを確認。
    4. unlock 後に Run intent shaping を実行し、policy / envelope / response が更新されることを確認。

    Operational rule

    The operator boundary is stored as an internal workspace identity created from the entered operator identity. If the session is locked, a mismatched operator identity or parallel windows are rejected before request shaping.

    A local operator lock keeps a single-window workflow active per account session, so parallel windows in the same browser profile are rejected while shaping output.

    This is not a generative chatbot replacement. It is a protocol cockpit where humans become the actor while the interface provides filtered, structured intent output.

    This is a static identity gateway: endpoint behavior and contracts are visible at a glance, and execution is always operator-mediated.
